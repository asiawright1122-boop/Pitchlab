# PitchLab:多渠道推送 + 自建精简 Agent(设计)

> 回应两个需求:①多渠道订阅推送(扫码绑定 TG/Discord/微信)②自己写精简 Agent
> 时间:2026-05-29

---

## 1. 为什么多渠道推送要自建(Hermes 不适合)

| | Hermes/OpenClaw | PitchLab 需要 |
|---|---|---|
| 用户模型 | **单用户**(你的私人助理) | **多租户**(海量订阅用户) |
| 渠道用途 | 助理在你的 TG 上跟你聊 | 产品向用户批量推送 + 绑定管理 |
| 缺的能力 | 用户账户/订阅/偏好/去重/限流/计费 | 全都要 |

> 结论:多渠道**订阅推送**是 SaaS 通知系统,不是个人 agent 的渠道。**必须自建轻量"通知/订阅服务"。**

---

## 2. 通知 / 订阅服务设计

```text
Web 账户 ──绑定──► [扫码/deep-link + 一次性 token] ──回调──► 绑定 chat_id 到账户
                                                          │
用户偏好(订阅哪些联赛/市场/信号阈值/推送时段)            │
                                                          ▼
事件(新预测/value信号/赛果/战绩更新) ──► 通知编排(去重+限流+合规措辞) ──► 多渠道发送器
                                                                          ├─ Telegram Bot
                                                                          ├─ Discord Bot/Webhook
                                                                          ├─ 微信公众号/企业微信
                                                                          └─ 邮件/Web Push(兜底)
```

### 各渠道实现(诚实)
| 渠道 | 难度 | 绑定方式 | 注意 |
|---|---|---|---|
| Telegram | 🟢 | Bot deep-link `t.me/bot?start=<token>` 二维码 | 最简单,首发推荐 |
| Discord | 🟢 | Bot + OAuth2 / 频道 webhook | 社区氛围好 |
| 微信 | 🟡 | **公众号**带参数二维码 + 模板消息;或**企业微信**机器人 | 个人微信无开放 API;模板消息有规范限制 |
| 邮件 / Web Push | 🟢 | 标准 | 兜底,人人可达 |

### 合规措辞(D7′)
推送内容 = **概率 / 分析 / value 信号 / CLV 战绩**;**禁用**"稳赚/必中/跟单暴富";给"仓位建议"非"代下注"。

## 3. 自建精简 Agent(领域编排器)

### 3.1 定位:不是通用 agent,是"足球量化编排层"
- 90% 确定性流水线(采集→清洗→特征→概率→回测)= **可重放代码**,不是自由 agent
- 仅 2–4 个 LLM 节点:文本抽取 / 报告 / (可选)假设发现 / (可选)残差修正
- 自己写 = 可控 / 低成本 / 可回测 / 可商业化

### 3.2 组成(代码量可控)
```text
pitchlab-agent(自建,精简):
  ├─ scheduler        # cron / 任务触发(可用成熟库,不自造调度内核)
  ├─ task-graph       # 任务编排:定义 L1→L7 的 DAG,幂等+重试+快照
  ├─ tool-registry    # 把量化能力注册为可调用工具(也可同时暴露为 MCP)
  ├─ llm-nodes        # 文本抽取 / 报告生成(便宜模型+缓存+预算护栏)
  ├─ skill-store      # 借鉴 Hermes:把"联赛/情境分析流程+配置"存为可复用 skill(md/json)
  ├─ memory           # 借鉴 Hermes:名称映射/特征有效性/校准历史(DB + 文件)
  └─ run-log          # 每次运行的输入快照/版本/产出(回测+审计)
```

### 3.3 借鉴 Hermes 范式但不背运行时
| Hermes 范式 | 我们怎么用 | 不同点 |
|---|---|---|
| skill 自沉淀 | 流程/配置存为可复用 skill | 沉淀流程不学概率 |
| MEMORY/USER 文件 | 映射/校准/偏好持久化 | 多租户存 DB |
| cron 调度 | 同 | 轻量库即可 |
| 多渠道 | **自建多租户通知服务** | 非单用户助理 |
| execute_code/MCP | 工具可选暴露为 MCP | 起步内部直调,按需 MCP |

### 3.4 为什么不直接用 Hermes(再次确认)
- 单用户 ≠ 多租户(推送需求决定)
- 重 + 成本不可控($1.3M 教训)
- 回测要求确定性,通用 agent 自由度反成负担
- **代码量:自建精简编排层并不大**,且完全贴合 + 可商业化
