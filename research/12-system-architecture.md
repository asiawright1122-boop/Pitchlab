# PitchLab 系统架构:单体 vs 微服务 / 订阅分层 / 多租户隔离

> 回应:Agent 用单服务还是微服务?用户订阅、会员分层、用户隔离怎么设计?
> 时间:2026-05-29

---

## 0. 关键澄清:Agent 是"全局单例",不是"每用户一个"

> 量化研究引擎跑**一次**,产出预测/信号/CLV = **全体共享的平台数据**。
> 用户差异 = 「能看到什么(会员分层)」+「订阅了什么(推送)」。
> ❌ 绝不 per-user 跑 agent(Hermes 单用户模式,又贵又错)。

这点决定了:**计算与数据是共享的,隔离的只是"用户域数据"和"访问权限"**。

---

## 1. 单体 vs 微服务 → 选「模块化单体 + 独立 Worker」

**起步不用微服务**(小团队/早期,微服务是过度工程:运维重、分布式复杂、schema 演进慢、拖慢迭代)。瓶颈在"找 edge + 产品",不在"扛百万并发"。

### 推荐:模块化单体(Modular Monolith)+ 1 个独立 Worker

```text
┌─────────────────────────────────────────────────────────┐
│  Web/API 服务(面向用户)                                  │
│   ├─ Auth / 账户                                          │
│   ├─ 订阅 & 计费(会员分层 entitlements)                  │
│   ├─ 看板(比赛/预测/value/公开 CLV 战绩)                 │
│   ├─ 渠道绑定(扫码 TG/Discord/微信)                      │
│   └─ 通知/订阅服务(多租户推送)                           │
├─────────────────────────────────────────────────────────┤
│  Agent/Worker 服务(全局单例,后台)                       │
│   └─ cron + 任务图:L1采集→…→L7结算  +  回测              │
├─────────────────────────────────────────────────────────┤
│  共享 packages:core(七层量化)/ db / llm / types         │
├─────────────────────────────────────────────────────────┤
│  PostgreSQL(单库,用户域行级隔离)  +  (可选)Redis 队列   │
└─────────────────────────────────────────────────────────┘
```

**为什么这样分:**
- **Web/API 与 Worker 分开**:两者**伸缩特性不同**(Web 应对请求峰值;Worker 是长任务/定时),进程隔离避免互相拖累
- **模块边界清晰**:七层 + 通知 + 订阅各为独立模块/包 → **将来要拆微服务也容易**(先单体,后按需提取)
- **共享 DB**:平台数据(赛事/预测/赔率/指标)全局共享读;用户数据按 user_id 隔离

### 何时才拆微服务?
当某模块有**独立伸缩/独立故障域**需求时再提取,例如:
- 实时赔率高频采集(独立节流/高频)
- LLM 抽取 Worker(独立成本/并发控制)
- 通知发送(独立队列/重试)

> 原则:**先模块化单体,边界画好;微服务等"被逼到"再上。**

## 2. 订阅 & 会员分层(Entitlements)

### 2.1 分层模型
不要把 tier 硬编码进逻辑,用 **entitlements(权益)映射**:

```text
Plan ──► Entitlements(权益集合)──► 在 API/UI 中间件统一鉴权

示例分层:
  Free      公开战绩查看 / 1-2 个联赛 / 延迟信号 / 无推送
  Pro       全联赛 / 实时 value 信号 / devig+EV+Kelly 工具 / 多渠道推送
  Premium   + 历史 CLV 深度 / 自定义筛选 / 提前推送窗口
  API/B2B   程序化访问 / 配额 / SLA
```

### 2.2 权益闸门(enforcement)
- 在 **API 中间件**统一校验 `user.entitlements`(不在前端做安全判断)
- 资源粒度:联赛范围、信号实时性、推送渠道数、历史深度、API 配额
- 计费:Stripe(或合规替代,见 §11 合规)+ webhook 同步订阅状态 → 更新 entitlements

### 2.3 数据模型(草案)
```text
users(id, email, ...)
plans(id, name, price, entitlements_json)
subscriptions(id, user_id, plan_id, status, period_end, provider_ref)
channel_bindings(id, user_id, channel, chat_id, verified_at)
user_subscriptions_topics(user_id, league_id/market, threshold)  # 订阅哪些
-- 平台共享数据(非用户隔离):
fixtures / odds_snapshots / predictions / metrics(CLV/ROI/Brier) / models
```

## 3. 多租户隔离(本质:用户域数据隔离,非每户一套系统)

### 3.1 隔离什么 / 共享什么
| 类别 | 隔离? | 例子 |
|---|---|---|
| 用户域数据 | ✅ 按 user_id 隔离 | 偏好、渠道绑定、订阅项、纸面跟单、收藏 |
| 平台数据 | ❌ 全局共享 | 赛事、赔率、预测、CLV 指标、模型 |
| 访问权限 | ✅ 按 entitlements | 能看哪些联赛/信号/历史深度 |

### 3.2 隔离实现(由轻到重)
1. **行级 scoping(起步推荐)**:所有用户域表带 `user_id`,数据访问层强制按当前用户过滤;**绝不信任前端**
2. **Postgres RLS(可选加固)**:行级安全策略,数据库层兜底防越权
3. **(仅当转 B2B 白标)** 加 `tenant_id` / schema-per-tenant —— **现在不做**,等真有机构客户再上

### 3.3 安全要点
- 鉴权(身份)与授权(权益)分离,均在服务端
- 用户域数据查询一律注入 `user_id` 约束(ORM/数据访问层统一封装,防遗漏)
- 推送绑定 token 一次性 + 过期;chat_id 视为敏感数据(隐私合规)

## 4. 技术选型落地(对齐既有决策)
- Web/API:Next.js(看板 + API)
- Worker:Node/TS 进程(cron + 任务图);量化重计算可调 Python(Dixon-Coles/ML)经子进程或服务边界
- DB:PostgreSQL + Prisma;(可选)Redis 做队列/缓存
- 部署:起步**单仓 monorepo,两个可部署单元(web + worker)**;容器化便于后续拆分
